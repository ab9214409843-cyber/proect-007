## Найдено при комплексной проверке 2026-06-10 (закрывается в Этапе 08_01)
Мелочи низкого приоритета из ревизии этапов 1–4. Сейчас не критично (RLS защищает данные),
но к финальной вычитке закрыть:

- [ ] **Обработка ошибок БД в server actions.** `updateEventStatus` и `deleteEvent`
  (`app/src/app/(cabinet)/events/actions.ts`) молча игнорируют возможную ошибку записи — если
  сохранение не пройдёт, пользователь этого не увидит. Сделать как в `createEvent`/`updateEvent`:
  проверять `error` и показывать понятное сообщение.
- [ ] **Валидация дат в форме мероприятия.** Сейчас можно указать дату окончания раньше даты начала.
  Добавить проверку «дата окончания не раньше даты начала» (`EventForm.tsx` / `events/actions.ts`).
- [ ] **Нумерация этапов в заглушках карточки мероприятия** — если не исправили на этапе 5:
  в `app/src/app/(cabinet)/events/[id]/page.tsx` подписи «Документы — этап 6, Заметки опыта — этап 7»
  не совпадают с дорожной картой (документы = этап 7, опыт = этап 8).
- [ ] **Перед публичным запуском** (дубль напоминания из CLAUDE.md): включить подтверждение email,
  включить защиту от утёкших паролей (Supabase advisor предупреждает), пересмотреть минимальную
  длину пароля (сейчас 6 символов).
